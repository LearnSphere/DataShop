DataShop - Dataset Import Tools

The DataShop Dataset Import Toolset can be used to verify and import tab-delimited data from text files into the DataShop repository.

The Dataset Verification Tool (DVT)

The Dataset Verification Tool can be used by researchers to check the format of dataset files they wish to import into the DataShop respository.

Specifications

The DVT accepts tab-delimited text files as input. The format of the file mimics that of the files produces through a DataShop export, with a few differences.

The following column headings are required in the dataset import text file:

  • Anon Student Id
  • Session Id
  • Time
  • Student Response Type
  • Tutor Response Type
  • Level
  • Problem Name
  • Step Name
  • Outcome
  • Selection
  • Action
  • Input

Optional column headings include:
  • Time Zone
  • Student Response Subtype
  • Tutor Response Subtype
  • Attempt At Step
  • Feedback Text
  • Feedback Classification
  • Help Level
  • Total # Hints
  • Tutor Response Subtype
  • Condition Name
  • Condition Type
  • KC
  • KC Category
  • School
  • Class
  • CF

Column Heading Descriptions

While most of the column headings listed above are self-explanatory, some require additional description.

'Level' represents a Dataset Level. An example of the correct use of this column heading could be 'Level(Unit)', where 'Unit' is the dataset leve name. The Level column should always be of the format Level(dataset_level_name). If a dataset level name is not included, the name will default to "Default".

Condition Name and Condition Type should always be paired with each other, even if a condition does not have a condition type. The DVT will produce and error if these two columns are not paired together.

KC represents a Knowledge Component. An example of the correct use of this column heading could be 'KC(Area)', where 'Area' is the skill model name for that knowledge component. The KC column should always be of the format KC(skill_model_name). If a skill model name is not included, the name will default to "Default".

KC Category represents a Knowledge Component Category. An example of the correct use of this columnd heading could be 'KC Category(Area)', where 'Area' is the skill model name for that knowledge component. The KC Category column should always be of the format KC Category(skill_model_name). If a skill model name is not included, the name will default to "Default".

Additionally, KC and KC Category must always be paired with each other, in the same way that Condition Name and Type must be paired together.

CF represents a Custom Field. An example of the correct use of this column heading could be 'CF(Factor or add-m)', where 'Factor or add-m' is the name for that custom field. The CF column should always be of the format CF(custom_field_name). If a custom field name is not included, the name will default to "Default".

Additional Requirements

The DVT expects the dataset column headings to be in a particular order. Placing columns in unexpected locations can cause the DVT to fail during processing. The expected order of column headings is listed below:

  1. Anon Student Id
  2. Session Id
  3. Time
  4. Time Zone
  5. Student Response Type
  6. Student Response Subtype
  7. Tutor Response Type
  8. Tutor Response Subtype
  9. Level()
  10. Problem Name
  11. Step Name
  12. Attempt At Step
  13. Outcome
  14. Selection
  15. Action
  16. Input
  17. Feedback Text
  18. Feedback Classification
  19. Help Level
  20. Total # Hints
  21. Condition Name
  22. Condition Type
  23. KC()
  24. KC Category()
  25. School
  26. Class
  27. CF()

Multiple Columns
The DVT allows for multiple columns for the following column headings:

  • Anon Student Id
  • Level()
  • Selection
  • Action
  • Input
  • Condition Name
  • Condition Type
  • KC()
  • KC Category()
  • CF()

For example, if a dataset file had multiple students, the correct column format would be: "Anon Student Id" [tab] "Anon Student Id". "Level()", "Selection", "Action", "Input", and "CF()" follow this specification as well. If a dataset file has multiple conditions, the column format would be: "Condition Name" [tab] "Condition Type" [tab] "Condition Name" "Condition Type". KC and KC Category follow this model as well.

The only input required for the DVT is the dataset file name you wish to process. Your input file should contain ALL SAMPLES you wish to import into the repository. Results from the DVT processing are printed to the console and to an output text file called "results.txt".

The Dataset Import Tool (DIT)

The Dataset Import Tool can be used to easily import tab-delimited data into the DataShop respository. This tool is currently available only to DataShop technical staff.

Specifications

The DIT accepts tab-delimited text files as input. The format of the file mimics that of the files produces through a DataShop export, with a few differences. The DIT relies on the DVT to check the formatting of the input file. If the DVT finds any errors during processing the DIT exits. Errors from processing are printed to the console as well as "results.txt".

Running the Dataset Impor Tool

Use of the Dataset Import Tool requires a filename argument from build.properties called "importdata.filename" and a dataset name argument called "importdata.dataset.name". After setting this up, run the ant target "runImportTool". If you wish to run the DatasetVerificationTool standalone, use "runImportVerify". This ant target requires the same build.properties arguements as runImportTool.
